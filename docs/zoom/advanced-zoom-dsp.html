解析の順番

いきなり逆アセンブリに入るより、まず TXE1 と PRME でユーザー向けの意図を把握し、そのあと INFO の DSP load を見て、最後に DATA のコードと定数テーブルに入る方がはるかに速いです。

UI 名と係数テーブルが結びつく瞬間に、逆アセンブリのノイズが意味のある設計情報に変わります。
Resource Constraints

DSP load は「ただの数字」ではなく、アルゴリズムのコスト構造を示すヒントです。

高くなりやすい要因

長い FIR、複数 delay line、高品質な補間、nonlinear waveshaping、oversampling、複雑な modulation。

低く抑えやすい要因

少数の biquad、単純な gain、軽い envelope follower、短い buffer。

読み方

似た effect で DSP% が大きく違うなら、どこかに補間・追加 filter・多段構造の差があると考えます。

Coefficient Tables

テーブルは「つまみ位置をそのまま数式に入れない」ための設計資産です。

なぜ table を持つのか

ユーザーのノブ位置 0-10 は、必ずしも周波数や gain の線形変化に対応しません。聴感上気持ち良い変化にするには、log スケールや経験的なカーブが要るので、事前計算テーブルが便利です。

knob position -> perceptual mapping -> coefficient set

Comp での例

sensitivity table は threshold や detector gain に近い役割、tone table は shelving / filter 係数、attack table は包絡追従の time constant に対応している可能性が高いです。

Runtime Architecture

effect コードは「初期化」「パラメータ変更」「音声処理」に分かれていると読むと整理しやすいです。

典型的な分割

init() param_edit() process_block() or process_sample()

init はメモリや係数の初期化、param_edit はつまみ変更時の再計算、process はリアルタイムの本体です。

逆アセンブリでの探し方

文字列や symbol table が残っているなら、edit, init, on/off, main callback 相当の関数名から当たりを付けます。残っていない場合でも、テーブル参照の頻度や buffer へのアクセスパターンで役割を推定できます。

TI C6000 View

TI C6000 系を意識すると、なぜ table と固定的な構造が多いかが見えてきます。

VLIW 的発想

並列実行しやすい定型処理は有利です。filter や MAC の繰り返しは相性が良いです。

メモリアクセス

長い FIR や複数 buffer は計算だけでなくメモリアクセスでもコストを払います。

実装文化

テーブル駆動、事前係数計算、明確な state struct は DSP 実装では自然な書き方です。
